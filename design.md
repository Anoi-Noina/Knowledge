# 企業マスタ名寄せツール 要件定義書

> 本書は最初に合意した要件であり、PoC実装の過程で一部は簡略化・未実装のまま据え置かれている。**実装済みの正確な挙動・固定パラメータ・本書からの逸脱点は [`docs/superpowers/specs/2026-07-11-nayose-design.md`](docs/superpowers/specs/2026-07-11-nayose-design.md) の8章「実装仕様」を参照。** プログラムを再現する場合は本書ではなく詳細設計書8章を正とすること。
>
> **本書・詳細設計書はあくまでPoC（ダミーデータ）を前提とした設計であり、実データへの適用を保証するものではない。** 正規化ルール・ブロッキング方式（詳細設計書8.1〜8.2節）は実データの表記ゆれ実態を見ないまま決めた暫定仕様であるため、実データのテーブル構造（カラム構成・法人番号の有無・表記ゆれパターン等）が判明した時点で、それをもとに設計を練り直す必要がある。**実データのカラム一覧を受領した際は、それをそのまま設定ファイル（JSON）に機械的に落とし込むのではなく、必ず開発者へのヒアリングを行った上で設計を見直すこと。**（詳細は11章）

## 1. 背景・目的

企業マスタ、事業所マスタ、企業部署マスタにおいて、実体は同一企業であるにもかかわらずコードが分かれているデータが存在する。
これらに対し、名寄せの手がかりとなる「名寄せキー（統合ID）」を付与し、後続の人手による整理作業を支援する。

## 2. 対象データ

- 企業マスタ
- 事業所マスタ
- 企業部署マスタ

3つのマスタはそれぞれ別スクリプト実行（または同一スクリプトへの異なる設定）で処理する。
マスタごとに「どのカラムを名寄せキーとして使うか」を**リストで指定できる**設計とする。

例：
```yaml
company_master:
  file: company.csv
  key_columns:
    - company_name
    - address
    - phone_number
    - corporate_number   # 法人番号（あれば最優先で利用）

department_master:
  file: department.csv
  key_columns:
    - company_name
    - address
```

## 3. 入力

- 形式：CSV / Excel ファイル
- 各マスタごとに、名寄せに使うキー列を設定ファイル（YAML/JSON等）またはスクリプト引数で指定可能にする
- 使用が想定されるキー情報の例：
  - 企業名
  - 住所
  - 電話番号
  - 郵便番号
  - 法人番号（存在する場合は最優先キーとして利用）

## 4. マッチングロジック

- 名寄せキーごとに正規化処理を行う（表記ゆれ対応）
  - 企業名：全角/半角統一、法人格表記の揺れ吸収（(株)/株式会社 等）、空白除去 等
  - 住所：番地表記ゆれ、都道府県・市区町村の分割/統合 等
  - 電話番号・郵便番号：ハイフン有無の統一 等
- レコード同士を1件ずつ比較し、類似度スコアを算出（あいまいマッチング）
- **名寄せIDは観点ごとに複数列で付与可能とする**（例：名前ベースID、住所ベースID、総合ID など、最大3種類程度を想定）
- しきい値による自動判定（高スコア＝自動でID付与）を行うが、**最終的な整理・確定は人が行う**前提とし、判定根拠（スコア・マッチしたキー）も出力に含める

## 5. 出力

- 元データの全カラム + 付与した名寄せID（複数列）+ マッチングスコア・根拠情報
- 形式：CSV / Excel（入力と合わせる）

## 6. 実行方式

- 一度きりのバッチ処理（Pythonスクリプトを都度実行する形）
- 定期実行やGUIは対象外（将来的な拡張の余地は残す）

## 7. 制約事項

- 外部API・外部サイトへの企業情報照会は一切禁止（業務利用データのため）
- 処理はすべてローカル環境・保有データの範囲内で完結させる
- 利用ライブラリはローカルで完結するもの（例：あいまい文字列マッチング用ライブラリ等）に限定し、ネットワークアクセスを伴うものは使用しない

## 8. 技術スタック・アーキテクチャ方針

- 言語：Python
- 名寄せエンジン：**Splink**（MITライセンス、英国司法省開発のOSS）を採用
  - DuckDBバックエンドで完全ローカル動作（ネットワーク通信・外部API送信なし）→ 外部API禁止の制約に抵触しない
  - 60万件規模でも高速処理可能（同程度規模で実績あり）
  - ブロッキングルール（都道府県一致、郵便番号上3桁一致 等）により総当たり(O(N²))を回避
  - カラムごとに比較方式（完全一致 / レーベンシュタイン距離 / Jaro-Winkler類似度など）を指定可能
    → 「キーのtypeごとに比較方法を変える」という要望をそのまま実現できる
  - 正解ラベル付き教師データなしで、EMアルゴリズムにより教師なしでマッチング精度を学習
  - マッチしたペアをクラスタリングし、**候補グループID（cluster_id）を自動生成** → 名寄せIDの採番方式として採用
    - 名前ベース・住所ベースなど観点を変えて複数回実行し、それぞれ候補IDを列として出力
    - 総合的な統合判断はプログラム側で行わず、候補IDと元コードを人が見比べて最終確定する

### アーキテクチャ概要（3マスタ共通・独立処理）

1. 入力（CSV/Excel）読み込み（文字列型を厳密に維持）
2. カラム正規化（全角半角統一、法人格表記ゆれ吸収、ハイフン統一 等）
3. ブロッキングルール適用（比較対象ペアの絞り込み）
4. Splinkによる類似度スコアリング・マッチ判定
5. マッチしたペアをクラスタリング → 候補グループID付与
6. 元データ + 候補グループID + マッチ根拠（スコア等）を出力（CSV/Excel）
7. 人が候補IDを見ながら最終的な名寄せを確定

## 9. 決定事項（旧：未確定事項）

| 項目 | 決定内容 |
|---|---|
| データ件数規模 | 各マスタ最大60万件程度を想定。総当たり不可のため必ずブロッキングを実施 |
| ブロッキング方針 | 都道府県一致・郵便番号上3桁一致 等、粗いキーで事前グルーピングしてから比較 |
| 3マスタ間の関係性 | 各マスタ内で独立処理（マスタを跨いだ紐付けは対象外） |
| 名寄せIDの採番方式 | プログラムでは確定IDを出さず、観点別（名前ベース／住所ベース等）の「候補グループID」を自動採番し、元コードと並べて出力。最終確定は人が目視で行う |
| 総合IDの統合ロジック | プログラム側では自動統合しない（候補IDを複数列で出すのみ） |
| 法人番号の扱い | 存在有無は実データ未確認。**カラムとして存在する場合は完全一致による最優先キーとして扱う**（あいまいマッチ不要な確定キーのため）。存在しない場合は名前・住所ベースの通常フローで進める。key_columns への追加のみで対応でき、設計変更は不要 |
| 企業名の表記ゆれ | 法人格表記ゆれ（(株)/株式会社 等）と完全一致の両方が混在。あいまい一致（Jaro-Winkler/Levenshtein）は完全一致ケースも自動的に高スコアとなるため両対応可能 |
| 類似度のしきい値方針 | 緩め寄り。可能性が少しでもあれば候補として出力し、最終判断は人に委ねる（見逃し(false negative)より、人の確認手間が増える方を許容） |
| 文字コード／型 | UTF-8前提（Shift_JIS読み込みにも対応）。電話番号・郵便番号・法人番号は頭ゼロ落ち防止のため文字列型で厳密に扱う |

## 10. 残タスク（実装前に確認できるとより精度が上がるもの）

- 実データでの法人番号カラムの有無確認
- 実データでの企業名表記ゆれサンプルの確認（正規化ルールのチューニング用。ただし未確認でも実装・着手は可能）

## 11. 詳細設計時に必要な入力情報（テンプレート）

詳細設計（正規化ルール・ブロッキングルール・Splink設定の具体化）を行う際は、以下を3マスタそれぞれ分提供する。

- **カラム名一覧**
- **各カラムの役割**（企業名／住所／電話番号／郵便番号／法人番号／識別子（元コード）／その他）
- **サンプル行 2〜3件**（実データではなく、表記形式が分かるダミー値でOK）

提供形式は、CSV/Excelファイルのアップロード、またはテキストで下記のように記載する形のどちらでもよい。

**実データのカラム一覧を受け取った時点でやること（重要）:**

1. カラム一覧を機械的に既存の設定ファイル（`configs/*.json`）のテンプレートへ当てはめて終わらせない。カラム名だけでは判断できない項目（法人番号カラムの実在有無、企業名の表記ゆれの実態、住所フォーマットの粒度、電話番号・郵便番号のフォーマット、想定件数規模など）を、**開発者に対して必ずヒアリングする**。
2. ヒアリングで確認すべき事項の例:
   - 法人番号カラムは存在するか。存在する場合、桁数・欠損有無・全角/半角混在の有無
   - 企業名の表記ゆれの実例（法人格表記以外に、略称・旧社名・全角半角混在等がどの程度あるか）
   - 住所のフォーマット（都道府県が別カラムか、番地の表記パターン、ビル名・部屋番号の有無）
   - 電話番号・郵便番号のフォーマット（ハイフンの有無、国番号の有無、桁数）
   - 各マスタの実データ件数（詳細設計書8.2節のブロッキング方式がPoCの簡易実装のままで良いか、都道府県・郵便番号によるブロッキングへの作り直しが必要かを判断する材料になる）
   - 名寄せ観点（perspectives）として企業名・住所以外に追加したいキーがあるか
3. ヒアリング結果をもとに、詳細設計書（`docs/superpowers/specs/2026-07-11-nayose-design.md`）8章「実装仕様」の該当箇所（特に8.1正規化ルール・8.2ブロッキングルール）を実データ向けに更新してから、設定ファイルを作成する。**PoC版の暫定仕様（先頭キー列の完全一致ブロッキング等）をそのまま実データに使うことを前提にしない。**

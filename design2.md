# 企業マスタ名寄せツール 詳細設計書（PoC）

作成日: 2026-07-11
更新日: 2026-07-18（8章「実装仕様」を追加。実装と本書の乖離を解消し、本書のみでの再現を可能にした）
元となる要件定義書: `design.md`（プロジェクトルート）

> 本設計は **PoC（Proof of Concept）** として実装する。実データではなく、`sample_data/` に用意するダミーCSV（企業マスタ・事業所マスタ・企業部署マスタの3種）で動作確認を行う。実データのカラム構成が判明した時点で、設定ファイルを差し替えるだけで実データに適用できることを狙う。

## 1. スコープ

- `design.md` の要件定義書に基づき、3マスタ（企業マスタ／事業所マスタ／企業部署マスタ）分の名寄せパイプラインを最初から一括で実装する。
- 3マスタは独立処理（マスタを跨いだ紐付けは対象外）。
- 3マスタで処理ロジック（正規化・ブロッキング・Splinkマッチング・クラスタリング・出力）は共通化し、マスタごとの差異は設定ファイル（JSON）で吸収する。

## 2. 全体アーキテクチャ

- 言語: Python。仮想環境・依存管理は `uv` を使用（`pyproject.toml`）。
- 名寄せエンジン: Splink（DuckDBバックエンド、完全ローカル動作）。
- 3マスタ共通のコアパッケージ `src/nayose_core/` を作成し、マスタごとの違いはJSON設定ファイルで指定する。
- 実行はマスタごとのJSON設定ファイルをCLI引数で渡す方式。

  ```
  uv run nayose.py --config configs/company.json
  uv run nayose.py --config configs/department.json
  uv run nayose.py --config configs/division.json
  ```

## 3. 設定ファイル（JSON）

マスタごとに1つのJSONファイルを用意し、CLI引数でパスを渡す。名寄せIDは観点（perspective）ごとに複数付与可能とし、1つの設定ファイル内で観点のリストを定義する。

```json
{
  "input": {
    "file": "sample_data/company.csv",
    "format": "csv",
    "encoding": "utf-8"
  },
  "id_column": "company_code",
  "corporate_number_column": "corporate_number",
  "perspectives": [
    { "name": "name_based", "key_columns": ["company_name"] },
    { "name": "address_based", "key_columns": ["address"] },
    { "name": "comprehensive", "key_columns": ["company_name", "address", "phone_number"] }
  ],
  "output": {
    "file": "output/company_result.csv"
  }
}
```

- `input.format`: `csv` または `excel`
- `input.encoding`: `utf-8` または `shift_jis`（読込失敗時は utf-8 → shift_jis の順にフォールバック）
- `id_column`: 元データの識別子カラム名
- `corporate_number_column`: 法人番号カラム名（存在しない場合はキー自体を省略可）。存在する場合は完全一致による最優先確定キーとして扱う
- `perspectives`: 観点ごとの名前と、その観点で使う `key_columns` のリスト
- `output.file`: 出力ファイルパス（拡張子は入力形式に合わせる）

## 4. 処理フロー（1マスタあたり）

1. 入力読込。CSV/Excel、utf-8/shift_jis対応。電話番号・郵便番号・法人番号・識別子カラムは常に文字列型で読み込み、先頭ゼロ落ちを防止する。
2. 正規化。カラム名に含まれるキーワードで正規化関数を自動選択する（詳細・完全な対応表は8.1節）。
   - 企業名: 全角/半角統一、法人格表記ゆれ吸収（(株)/株式会社 等）、空白除去
   - 住所: 丁目・番地・号などの表記ゆれ吸収（**都道府県・市区町村の分割/統合は未実装**、8.1節参照）
   - 電話番号・郵便番号: 数字以外の文字を除去
3. `corporate_number_column` が設定に存在する場合、まずその完全一致でグルーピングする（あいまいマッチ不要の最優先確定キー）。
4. 各 `perspectives` ごとに以下を実行する。
   - ブロッキングルール適用（**実装は key_columns の先頭列の正規化後の完全一致のみ**。都道府県・郵便番号によるブロッキングは未実装。詳細は8.2節）
   - Splinkによる類似度スコアリング・EMアルゴリズムでの教師なし学習
   - しきい値判定（緩め寄り。見逃し防止を優先し、可能性があれば候補として残す。実装値は8.3節）
   - マッチしたペアをクラスタリングし、候補グループID（cluster_id）を付与
5. 元データの全カラム + 観点ごとの候補グループID列（例: `name_based_cluster_id`, `address_based_cluster_id`, `comprehensive_cluster_id`）+ マッチスコアを1つの出力ファイルにまとめて出力する（**「マッチしたキー」は出力しない**。要件定義書からの逸脱点。詳細は8.5節）。
6. 最終的な統合判断はプログラム側では行わず、人が候補IDと元コードを見比べて確定する。

## 5. エラーハンドリング

- 設定JSONのバリデーション: 必須キー欠落、指定カラムが入力データに存在しない場合はエラー終了し、分かりやすいメッセージを表示する。
- 文字コード読み込み失敗時は utf-8 → shift_jis の順にフォールバックを試行する。
- Splink実行中の異常（対象データ0件、ブロッキング結果が空 等）は警告ログを出し、可能な範囲で処理を継続する。

## 6. テスト方針（PoC）

- `sample_data/` に3マスタ分のダミーCSVを作成する（表記ゆれ・同一企業の重複を含む数十行程度）。
- `pytest` で正規化関数（全角半角、法人格表記ゆれ、ハイフン統一等）の単体テストを行う。
- ダミーデータを使ったEnd-to-Endテストを行い、設定ファイル→出力ファイルまで実行して、想定される候補グループIDが一致することを確認する。

## 7. 制約事項（要件定義書より踏襲）

- 外部API・外部サイトへの企業情報照会は一切禁止。処理はすべてローカル環境・保有データの範囲内で完結させる。
- 利用ライブラリはローカルで完結するものに限定し、ネットワークアクセスを伴うものは使用しない。

## 8. 実装仕様（完全再現用パラメータ）

本章は、本書のみを見てプログラムを再実装した場合に実装と同じ挙動になることを目的とし、コード上の固定値・分岐条件をそのまま記載する。7章までの記載と食い違う場合は本章が優先する。

### 8.1 正規化ルール（列名によるディスパッチ）

`normalize_column(column_name, series)` は、列名を小文字化した上で以下のキーワードを**部分一致**で判定し、正規化関数を自動選択する（設定ファイルでの明示指定はできない）。上から順に判定し、最初に一致したものを採用する。

| 列名に含まれる文字列 | 適用関数 | 処理内容 |
|---|---|---|
| `name` | `normalize_company_name` | NFKC正規化 → 下記の法人格表記を全て除去 → 空白（`\s+`）を除去 → strip |
| `address` | `normalize_address` | NFKC正規化 → `丁目`→`-`、`番地`→`-`、`番`→`-`、`号`→削除 → 各種ダッシュ記号（`‐－ー―−`）を`-`に統一 → 連続する`-`を1つに圧縮 → 空白を除去 → 前後の`-`とstrip |
| `phone` / `tel` / `postal` / `zip` のいずれか | `normalize_phone_or_postal` | NFKC正規化 → 数字以外の文字を正規表現`\D`で全除去（ハイフンに限らず記号・空白も除去） |
| 上記いずれにも該当しない | `_normalize_generic` | NFKC正規化 → strip のみ |

- 判定は部分一致のため、例えば `company_name_kana` も `name` を含み企業名正規化の対象になる。列名の命名によって挙動が変わる点に注意。
- 除去対象の法人格表記リスト（`_CORPORATE_SUFFIXES`、完全一致による文字列置換）:
  `株式会社`, `（株）`, `(株)`, `有限会社`, `（有）`, `(有)`, `合同会社`, `（合）`, `(合)`, `合資会社`, `合名会社`
- **都道府県・市区町村の分割/統合は未実装。** 要件定義書4章の記載より後退している（理由: PoCの対象データに都道府県が別カラムで存在せず、住所文字列からの機械的な分割は表記ゆれ・誤分割のリスクが高く、PoC範囲では見送った）。実データ適用時に再検討する（9章参照）。

### 8.2 ブロッキングルール（実装値）

`build_blocking_rules(key_columns)` は、**`key_columns[0]` を正規化した列（`{key_columns[0]}_norm`）の完全一致のみ**をブロッキングキーとして使う（`splink.block_on` を1本だけ生成）。2列目以降のキー列はブロッキングには使われず、類似度スコアリングにのみ使われる。

要件定義書・本書がこれまで記載していた「都道府県一致・郵便番号上3桁一致 等」の粗いブロッキングは**未実装**。PoC規模（数十件）では性能問題が顕在化しないため、先頭キー列そのものの完全一致という単純な方式で代替した。

この結果、**正規化後の先頭キー列が1文字でも異なる表記ゆれのレコード同士は、比較対象ペアとして生成されず、あいまいマッチングが機能しない**（例: `name_based` 観点で企業名の一部だけが違う表記ゆれは拾えない）。実データ移行時の見直し対象（9章参照）。

### 8.3 Splink実行パラメータ（固定値）

| パラメータ | 値 | 備考 |
|---|---|---|
| 比較関数 | `cl.JaroWinklerAtThresholds` を全キー列に一律適用 | 列ごとに比較方式（完全一致／レーベンシュタイン等）を切り替える機能・設定項目は未実装 |
| `link_type` | `"dedupe_only"` | |
| マッチしきい値（predict・clustering 共通） | `DEFAULT_THRESHOLD = 0.1` | 「緩め」方針の具体値。match_probability 0.1以上のペアを候補として残す |
| `estimate_probability_two_random_records_match` の `recall` | `0.9` | ブロッキングルールは8.2節と同じものを使用 |
| `estimate_u_using_random_sampling` の `max_pairs` | `1e6` | |
| EM学習用ブロッキングルール | key_columns が1列: `"1=1"`（全件総当たり） / 2列以上: `blocking_rules[0]`（8.2節と同じ先頭列完全一致） | 分岐理由は9章に既存の記載あり（`_em_training_blocking_rule`） |

### 8.4 出力仕様（生成ロジックの詳細）

- `{perspective}_cluster_id`: `linker.clustering.cluster_pairwise_predictions_at_threshold`（しきい値0.1）が付与するクラスタID。マッチ相手が1件もいないレコードにも単独クラスタとしてIDが付与される（空欄にはならない）
- `{perspective}_match_score`: そのレコードが関与した全ペアの `match_probability` の**最大値**（レコードがleft/rightどちらに現れたペアも対象）。マッチ相手が1件もいない場合は `0.0`
- `corporate_number_group_id`: `corporate_number_column` の値をそのまま使い `f"corp_{値}"` という文字列を生成するのみ（全角/半角統一などの正規化はしない）。空文字は比較前に `NA` に変換されるため、出力上も空欄のまま
- **要件定義書4章にある「マッチしたキー（判定根拠）」の出力は実装されていない。** 出力されるのは `match_score` のみで、どのキー列がマッチに寄与したかは分からない

### 8.7 エラーハンドリングの正確な境界

5章の記載は概要であり、実際に捕捉されるのは`nayose.py`の`main()`が`ConfigError`をキャッチする場合のみ。

| 発生条件 | 例外 | `main()`での扱い |
|---|---|---|
| 設定ファイルが存在しない／必須キー欠落／`perspectives`未設定／指定カラムが入力データに存在しない | `ConfigError`（`config.py`, `nayose.py`） | 捕捉され `設定エラー: {メッセージ}` を標準エラー出力に表示し、終了コード1で終了 |
| 入力ファイル（CSV/Excel）自体が存在しない | `FileNotFoundError`（`io.py`） | **捕捉されない。** Pythonの生のトレースバックが出力され、異常終了する（「設定エラー」の体裁にはならない） |
| `input.format`/`output.file`の拡張子指定が`csv`/`excel`以外 | `ValueError`（`io.py`） | **捕捉されない。** 同上、生のトレースバックで終了 |
| 文字コード読み込み失敗（`utf-8`↔`shift_jis`） | なし（自動フォールバックで処理継続） | 例外は発生しない。フォールバックも失敗した場合は`UnicodeDecodeError`が捕捉されず異常終了 |

## 9. 今後の課題（PoC後）

- 実データのカラム名・役割・サンプル行の提供を受け、設定ファイル（JSON）を実データ用に作成する。**ただしカラム一覧を機械的に設定ファイルへ変換するのではなく、開発者へのヒアリングを行った上で本書8章（特に8.1正規化ルール・8.2ブロッキングルール）を実データ向けに練り直してから作成する（要件定義書`design.md` 11章参照）。** 本書8章の内容はPoC用ダミーデータを前提とした暫定仕様であり、実データへの適用を保証しない。
- 実データでの企業名表記ゆれサンプルを確認し、正規化ルールをチューニングする。
- 60万件規模のデータでの性能検証（ブロッキングルールの調整含む）。
- **単一キー列の観点（例: `key_columns` が1列のみの `name_based`）は、現状EM学習（パラメータ推定）を全件同士の直積（ブロッキングなし）で行っている。** これはSplinkの仕様上、EM学習時のブロッキング列と比較列が完全に一致すると推定対象がゼロになりクラッシュするための回避策（`src/nayose_core/matching.py` の `_em_training_blocking_rule`）。PoC規模のダミーデータでは問題ないが、60万件規模の実データにそのまま適用すると組み合わせ数が膨大になりハング・メモリ不足の恐れがある。実データ移行時は、EM学習用に比較列とは別の粗いブロッキング列（都道府県一致・郵便番号上3桁一致 等）を用意し、単一キー列観点でも直積を避けられるよう設計を見直す。

# kiritsu — 実装前レビューの徹底

## 目的
プログラム改修の依頼に対して、確認や修正の機会を与えずにいきなり実装を始めてしまうことを防ぐ。

## 手順

1. **要件の整理**
   - 依頼内容を読み、改修の目的・対象範囲・制約を整理する。曖昧な点があれば実装に入る前にユーザーに確認する。

2. **計画モードに入る**
   - `EnterPlanMode` ツールを呼び出す。この段階ではコードを一切変更せず、調査（既存コードの読み込み、影響範囲の把握など）のみ行う。

3. **計画書の作成**
   - 以下の項目を含む計画書をまとめる。
     - 目的・背景
     - 変更対象ファイル / モジュール
     - 具体的な変更方針・手順
     - 影響範囲・リスク
     - 未確定事項や確認したい点

4. **承認を得る**
   - `ExitPlanMode` を呼び出し、計画書をユーザーに提示する。
   - ユーザーが修正を求めた場合は、実装に入らず計画書を更新し、再度提示する。明示的な承認（例:「OK」「進めて」「この内容で」）が得られるまでこのループを続ける。

5. **承認後に実装**
   - 承認が得られて初めてコードの変更に着手する。
   - 実装中に計画から外れる判断が必要になった場合は、いったん手を止めてユーザーに確認する。

## 禁止事項

- 計画書の提示・承認を経ずに `Edit` / `Write` などでコードを変更しない。
- 「簡単そうだから」「明らかだから」という理由で本手順を省略しない。
